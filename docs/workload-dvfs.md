# Workload-Aware DVFS：动机与目标

## 动机

### 当前 DVFS 的盲区

动态电压频率调节（DVFS）是现代处理器节能的核心技术。Linux 内核提供了多种 cpufreq governor（如 ondemand、schedutil、powersave）来自动调整 CPU 频率。然而，这些 governor 有一个共同的盲区：**它们不区分工作负载的类型**。

ondemand 和 schedutil 的决策逻辑本质上是相同的：CPU 利用率高就升频，利用率低就降频。这个策略基于一个隐含假设——**CPU 忙的时候需要高性能，闲的时候可以省点电**。

但这个假设忽略了一个关键事实：**不同类型的工作负载对频率的敏感度完全不同**。

### 计算密集型 vs 访存密集型

考虑两种极端情况：

**计算密集型**（如矩阵乘法、编译器优化、视频编码）：CPU 的计算单元满负荷运转，每条指令都依赖上一条的结果。频率翻倍，吞吐翻倍；频率减半，性能减半。这类工作负载对频率高度敏感。

**访存密集型**（如数据库查询、图遍历、稀疏矩阵运算、Web 服务）：CPU 大部分时间在等待数据从内存到达。一次 DRAM 访问的延迟约 100 纳秒，而一个 3 GHz 的 CPU 周期只有 0.33 纳秒。在这 100 纳秒的等待里，CPU 无论运行在什么频率，都只能干等。频率翻倍，等待时间不变；频率减半，吞吐几乎不变。这类工作负载对频率几乎不敏感。

### 被浪费的能源

问题在于：**现实中大量工作负载是访存密集型的**。

当 governor 看到 CPU 利用率 100% 时，它会认为"这个任务需要最高性能"，于是把频率拉到最高。但如果这个任务其实是访存密集型的，那么从 800 MHz 提升到 3000 MHz，性能几乎没有变化，功耗却增加了数倍。

这不是理论推演。在数据中心的实际测量中，大量任务的"频率甜点"（sweet spot）远低于最大频率。有些任务在 30% 最大频率下就能达到 95% 的峰值性能。

### 功耗的物理现实

CPU 动态功耗遵循公式：**P ∝ V² × f**（功耗与电压平方成正比，与频率成正比）。

降频通常伴随降压。这意味着降频的节能效果是超线性的：

- 频率降到 70%，电压也降到 70%：功耗约降到 70% × (0.7)² ≈ **34%**，节能 **66%**
- 频率降到 50%，电压也降到 50%：功耗约降到 50% × (0.5)² ≈ **12.5%**，节能 **87.5%**

如果一个任务的甜点在最大频率的 30%，那么理论上可以节省超过 80% 的功耗，而性能损失不到 5%。

**但前提是：你得知道甜点在哪里。**

### 为什么 Governor 找不到甜点？

一个自然的想法是：让 governor 在运行时自己探索频率-性能曲线，找到甜点。但这在实践中几乎不可行：

1. **测量需要时间**。要准确测量某个频率下的性能，至少需要稳定运行几秒钟。生产环境的工作负载不会给你这样的窗口。
2. **频率切换有延迟**。cpufreq 的频率切换需要 10-50 微秒，频繁扫描会干扰正常工作。
3. **噪声难以过滤**。运行时的性能波动来自后台进程、中断、温度变化等多种因素，单次测量不可靠，需要多次采样取中位数。
4. **缺乏对照**。要验证频率设置是否真的生效（而不是被 turbo boost 覆盖），需要一个纯计算的对照基准。这在生产环境中无法插入。

## 目标

Workload-Aware DVFS 的核心思想是：**离线分析工作负载的频率特性，在线利用分析结果指导频率决策**。

### 目标一：识别工作负载类型

不同的访存模式对频率的敏感度不同：

- **顺序访问**（如数组遍历）：硬件预取器能预测访问模式，带宽利用率高，对频率 moderately 敏感
- **随机访问**（如链表遍历、哈希表查找）：预取器失效，每次访问都是 L3 miss，对频率几乎不敏感
- **混合模式**（如数据库索引查找）：介于两者之间

Workload-Aware DVFS 需要能区分这些模式，并分别测量它们的频率-性能曲线。

### 目标二：量化频率-性能关系

对于每种工作负载，需要回答：

- **甜点在哪里？** 能保持 95% 峰值性能的最低频率是多少？
- **平台期有多宽？** 从甜点往下，性能下降的曲线有多陡？
- **多核场景下呢？** 单个核心的甜点和 64 个核心同时运行的甜点可能差几百 MHz（因为内存带宽竞争）。
- **不同缓存层次呢？** 工作集在 L3 内 vs 溢出到 DRAM，甜点会跳变。

### 目标三：提供决策依据

一个频率数字不够，还需要：

- **置信区间**。通过 bootstrap 重采样，给出甜点的 95% CI。区间窄说明决策可靠，区间宽说明需要更多数据。
- **功耗节省百分比**。在甜点处运行，相对于最大频率能省多少电？这直接决定 ROI。
- **阈值敏感性**。如果把"95% 性能"放宽到"90% 性能"，能多省多少电？帮助决策者在性能和功耗之间权衡。

### 目标四：支持动态调整

工作负载的特征会随时间变化：

- 白天是 Web 服务（访存密集），晚上是批处理（计算密集）
- 数据库查询的工作集大小随数据增长而变化
- 容器化的微服务，不同 Pod 运行不同的工作负载

Workload-Aware DVFS 需要支持：

- **按核心设置**。不同核心运行不同工作负载，各自有各自的甜点。
- **按时间段切换**。定时任务在不同时段使用不同的频率策略。
- **按容器隔离**。Kubernetes 场景下，通过注解（annotation）为每个 Pod 指定频率约束。

### 目标五：与现有系统集成

Workload-Aware DVFS 不应该替代现有的 governor，而是增强它：

- **最简方案**：通过 `scaling_min_freq` / `scaling_max_freq` 把频率约束在甜点附近，让 governor 在"正确范围"内做细粒度决策。
- **中等方案**：自定义 userspace governor，读取离线分析结果，动态调整频率。
- **最灵活方案**：修改内核 governor（如 schedutil），在调度决策时考虑工作负载类型和甜点。

## 总结

Workload-Aware DVFS 的本质是：**用离线分析换取在线智能**。

离线阶段，通过系统化的基准测试，穷举不同频率、不同核心数、不同工作集大小下的性能数据，建立完整的"频率-性能-功耗"模型。

在线阶段，governor 不再盲目地"忙就升频"，而是知道"这个工作负载在 1800 MHz 就够用了，再高只是浪费电"。

最终效果：**在不损失性能的前提下，让每一瓦特的功耗都产生实际价值**。
